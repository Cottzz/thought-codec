# 代码与组件重构器

## 什么时候使用

当代码、脚本或自动化流程中出现可复用的重复模式，需要沉淀为模块、工具函数或规则时，使用本 Skill。

适合场景：

- 多个函数做了几乎相同的事情。
- 一个脚本混杂了数据处理、打印、渲染和副作用。
- Bug 修复暴露出可复用的防御性模式。
- 一次性代码应升级为库级组件。

## 输入

需要收集：

- 源文件或代码片段。
- 不允许改变的既有行为。
- 重复模式或痛点。
- 运行环境、语言、框架和测试约束。
- 目标形态：工具函数、类、包、hook、API、CLI 或 Skill 规则。

## 工作流程

1. 先检查现有代码，再提出抽象。
2. 识别重复、边界不稳、硬编码和副作用。
3. 通过测试或明确验证步骤保护既有行为。
4. 强制区分计算与表现：
   - 核心逻辑应返回数据。
   - UI、日志、打印和渲染应留在核心逻辑之外。
5. 提取最小可用的复用单元。
6. 只有在风险清楚时才替换调用点。
7. 记录接口、示例和边界。
8. 将可复用工程经验压缩为未来规则。

## 输出约定

返回：

1. `观察到的模式`
2. `抽象候选`
3. `行为保护计划`
4. `建议接口`
5. `实施步骤`
6. `测试或验证`
7. `可复用规则`

## Codex 使用说明

Codex 应保持修改范围清晰，遵循仓库风格，运行针对性测试，并避免不必要的大范围重构。不得覆盖用户的无关改动。

## Claude 使用说明

Claude 应在代码修改前帮助判断抽象边界、命名、取舍和规则提取。

## 边界

- 不因为两个片段看起来相似就强行抽象。
- 没有测试或明确验证前，不改变行为。
- 不把重要领域差异隐藏进过度通用的 helper。
- 抽取组件时不得混合计算与表现。

## 示例提示词

```text
使用代码与组件重构器。

来源：三个脚本都在解析产品评价 JSON，但各自硬编码路径并直接打印。
目标：抽取可复用解析器，将打印留在独立 CLI 层。
输出：接口、实施计划、测试和可复用规则。
```
